#### 1. نظام المسار الهندسي (Engineering Workflow Engine)

##### الفكرة الأساسية
استبدال إدارة المشاريع البسيطة بنظام مسار هندسي متكامل يتبع المراحل الفعلية لمكاتب الاستشارات الهندسية في الخليج. كل مشروع يمر بمراحل واضحة من التصميم المعماري حتى الموافقات الحكومية، مع نظام SLA تلقائي يراقب المواعيد.

##### النماذج الجديدة في قاعدة البيانات

| النموذج | الوظيفة |
|---------|---------|
| **WorkflowPhase** | 21 نوع مرحلة هندسية مع SLA، تتبع الاعتماديات، عداد الرفض |
| **ClientInteraction** | نظام اعتماد/رفض/تعليق من العميل يؤثر مباشرة على حالة المرحلة |
| **AuditLog** | سجل عمليات شامل يراقب كل تغيير في النظام |

##### المراحل الهندسية المدعومة (21 مرحلة)

**المعماري (7):** Sketch → Concept Design → Client Approval → Modifications → Preliminary → 3D Max → Final Drawings

**الإنشائي (4):** Soil Report → Foundation → Structural Calculations → Structural Drawings

**الخدمات MEP (5):** Electrical → Drainage → Water Supply → HVAC → Etisalat

**الموافقات الحكومية (4):** Municipality Submission → Municipality Approval → Civil Defense → SEWA/DEWA

**التعاقد (1):** Contract Signing

##### نظام الـ SLA (Service Level Agreement)

كل مرحلة عندها عدد أيام محدد:
- Sketch → 4 أيام
- Client Approval → 3 أيام
- Modifications → 4 أيام
- Preliminary → 6 أيام
- Municipality → 7 أيام
- Civil Defense → 7 أيام
- Etisalat → 4 أيام
- SEWA/DEWA → 6 أيام

**المنطق:**
- عند بدء أي مرحلة يبدأ العد التنازلي تلقائياً
- عند التأخير: تتحول الحالة للأحمر + إرسال إشعار للمدير
- الألوان: أخضر (في الوقت) / أصفر (تحذير، يوم متبقي) / أحمر (متأخر)

##### نظام الاعتماد المتبادل (Dependencies)

- ❌ لا يبدأ الإنشائي إلا بعد موافقة العميل على المعماري
- ❌ لا يتم تقديم البلدية بدون المخططات النهائية
- ❌ لا يتم إنشاء فاتورة إلا بعد إتمام المرحلة
- التنفيذ: عبر `dependsOnId` مع `checkAllDependenciesCompleted()`

#### 2. صفحة المشروع الواحد (Project Workspace)

##### الفكرة الأساسية
بدلاً من التنقل بين صفحات متعددة، كل بيانات المشروع متاحة في صفحة واحدة مع تبويبات ديناميكية حسب دور المستخدم. الموظف لا يفتح أكثر من صفحة واحدة.

##### المكونات المبنية (1466 سطر)

**الـ Sidebar الثابت (260px):**
- اسم المشروع، الرمز، الموقع
- اسم العميل وحالة الاشتراك
- Progress Bar للتقدم
- إحصائيات سريعة: الميزانية، المصروف، المهام، المراحل المكتملة
- زر "اسأل بلو" (يفتح المساعد الذكي في وضع المشروع)

**التبويبات الـ 9:**

| التبويب | المحتوى |
|---------|---------|
| **Overview** | تفاصيل المشروع، ملخص هندسي + مالي + مهام |
| **Architecture (المعماري)** | 7 مراحل مع SLA countdown + حالة dropdown |
| **Structural (الإنشائي)** | 4 مراحل مع نفس النمط |
| **MEP (الخدمات)** | 5 مراحل (كهرباء، صرف، مياه، تكييف، اتصالات) |
| **Government (الموافقات)** | 4 مراحل + عداد الرفض + سجل ملاحظات البلدية (MUN NOT) تراكمي |
| **Financial (المالية)** | قيمة العقد، الدفعات، جدول الكميات (BOQ)، الرصيد |
| **Site (الموقع)** | يوميات الموقع + العيوب والمخالفات |
| **Timeline (الجدول الزمني)** | عرض بصري لكل المراحل مع ألوان الحالة |
| **Interactions** | سجل تعاملات العميل (اعتماد/رفض/تعليق) مع إمكانية الرد |

##### ميزة MUN NOT (تراكمي)
هذا الحقل مصمم خصيصاً ليكون "تاريخي" وليس مجرد حقل. كل مرة يتم رفض المخططات من البلدية، يتم تسجيل الملاحظات مع عداد الرفض. المساعد الذكي (بلو) يستطيع تحليل هذا التاريخ لاقتراح حلول وتحسين جودة المخططات مستقبلاً.

##### نظام الـ Version Control للمستندات
- كل ملف في نظام المستندات عنده version (V1, V2, Final)
- تسجيل من قام برمج `uploadedById` وتاريخ الرفع
- حل مشكلة "طباعة نسخة قديمة بالخطأ" الشائعة في المكاتب

#### 4. نظام تفاعل العميل (Client Interaction)

##### الفكرة الأساسية
نظام بسيط وفعّال يسمح للعميل بالتفاعل مع مراحل المشروع:
- **اعتماد (APPROVE):** تحديث حالة المرحلة لـ "مكتمل" وفتح المرحلة التالية
- **رفض (REJECT):** زيادة عداد الرفض + تحديث حالة المرحلة
- **تعليق (COMMENT):** إضافة ملاحظات بدون تغيير الحالة

**التأثير المباشر:**
- حالة المرحلة تتغير فوراً
- المرحلة التالية تصبح جاهزة للبدء (لو dependencies مكتملة)
- سجل كامل في ClientInteraction + AuditLog

#### 5. تحسين المساعد الذكي (Blue AI)

##### الإضافات
4 أزرار اقتراحات سريعة في الأعلى:
1. **تحليل ملاحظات البلدية** - يجلب بيانات MUN NOT ويحلل أسباب الرفض المتكررة
2. **ملخص مالي** - يعرض ملخص الميزانية والدفعات والفواتير المستحقة
3. **حالة المشروع** - يعرض حالة المشروع والمراحل النشطة والمتأخرة
4. **مهام متأخرة** - يعرض المهام التي تجاوزت الموعد النهائي

##### آلية العمل
كل زر يرسل prompt مخصص للـ AI مع بيانات فعلية من قاعدة البيانات. الـ AI API في `/api/ai/route.ts` يقرأ السياق (`context`) ويجيب البيانات المناسبة قبل إرسالها للنموذج.

#### 6. إدارة أحمال الفريق (Workload Widget)

##### الفكرة الأساسية
كارت جديد على الـ Dashboard يعرض حمل عمل كل عضو في الفريق:

| البيان | الوصف |
|-------|-------|
| Active Tasks | عدد المهام النشطة (TODO + IN_PROGRESS + IN_REVIEW) |
| Overdue Tasks | عدد المهام المتأخرة |
| Completion Rate | نسبة إنجاز المهام المكتملة |
| Load Status | 🟢 ok (أقل من 4 مهام) / 🟡 moderate (4-10) / 🔴 overloaded (أكثر من 10 أو أي مهام متأخرة) |

#### 7. سجل العمليات (Audit Log)

##### الفكرة الأساسية
سجل مركزي لكل عملية تحدث في النظام:

- **أنواع الكيانات:** project, task, invoice, contract, workflow_phase, client_interaction
- **أنواع العمليات:** CREATE, UPDATE, DELETE, APPROVE, REJECT, SIGN, VIEW
- **البيانات المسجلة:** القيمة القديمة والجديدة، عنوان IP المستخدم، الطابع الزمني

##### مواقع التسجيل التلقائي
- تحديث حالة مرحلة هندسية
- إنشاء أو حذف مهمة مرتبطة بمرحلة
- اعتماد/رفض من العميل
- بدء/إيقاف مرحلة

### ⚡ ملاحظات تقنية

1. **التخزين:** المشروع يستخدم SQLite في بيئة التطوير لكن المنصة الأصلية تستخدم PostgreSQL. لازم التأكد من توافق الـ SQL (لم نستخدم أي PostgreSQL-specific features).

2. **الـ SPA Routing:** التطبيق يستخدم نمط SPA وليس file-based routing. كل الصفحات تُعرض من `page.tsx` عبر `switch(currentPage)`. هذا يعني لا يوجد `/dashboard/projects/[id]` URL فعلي - التنقل يتم بالـ state فقط.

3. **الـ Demo Mode:** النظام يعمل بدون قاعدة بيانات في "demo mode" (يستخدم placeholder data). هذا ممتاز للاختبار.

4. **الـ RTL Support:** الدعم الكامل للعربية مع `dir="rtl"` وترتيب عناصر البنية.

5. **الـ Dark Theme:** التصميم الافتراضي هو Dark theme مع `bg-slate-950` و`bg-slate-900`.
